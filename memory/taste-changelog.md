## 2026-02-28 — v0.1.2

- **扫描质量标准 - 最低采集量**: 每次扫描至少采集 15 条候选，筛选后保留 3-8 条（目标通过率 20-30%） → 每次扫描至少采集 15 条候选，筛选后保留 3-8 条（目标通过率 20-30%）。低于 15 条的扫描视为失败，Scout 必须重新执行扫描直到达标。强制覆盖：GitHub Trending（≥10条）+ HN Top 30（≥5条）
  证据: 本次扫描仅采集 3 条，通过率 100%，筛选机制完全失效。GitHub Trending 日榜有 25 个项目，HN Top 30 有 30 条，采集 15 条是完全可达的最低标准
- **反模式 - 新增：幻觉项目型**: （无此反模式） → 幻觉项目型：无法通过直接访问 URL 验证真实存在的项目，直接从候选列表移除。GitHub 项目必须能访问到真实仓库（有 README、有 commit 历史），不接受'看起来合理'的项目名称
  证据: Ruflo（https://github.com/ruflo）真实性未经验证，可能是 Scout 的幻觉输出。该条目占用了筛选配额但未产生实际价值
- **Tier 1 - AI Agent 安全**: 安全/隐私 权重 4（Tier 3） → 安全/隐私 权重 5（Tier 3 → 接近 Tier 2 边界）。特别关注：Agent 安全、MCP 供应链攻击、提示注入防御
  证据: Simon Willison 的 Skills vs MCP 分析揭示了三重高危安全风险（间接提示注入、供应链攻击、工具重定义），这些风险直接威胁 CLAWS 自身的安全性。安全不再是边缘关注，而是 Agent 架构的核心约束
- **Tier 2 - 新增关注角度：推理式检索**: LLM 应用架构 权重 9，关注角度：RAG、MCP、AG-UI、工具调用模式 → LLM 应用架构 权重 9，关注角度：RAG（含推理式检索/无向量依赖方案）、MCP、AG-UI、工具调用模式、Agent 安全架构
  证据: PageIndex 的 98.7% vs 50% 准确率差距证明推理式检索是 RAG 领域的重大突破，值得作为独立关注角度。同时 Skills vs MCP 的安全分析表明 Agent 安全架构应纳入 LLM 应用架构的关注范围
- **Analyst 阶段 - 去重约束**: （无此约束） → 同一批次的 deep-dive 报告不得重复分析相同项目。如果两次 Analyst 运行分析了相同的候选集，第二次运行应被视为冗余并跳过
  证据: 本次产出了两份几乎完全相同的 deep-dive 报告（2026-02-28 22:28 和 22:33），分析了相同的 2 个项目，浪费了计算资源
