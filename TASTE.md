## 核心审美原则

这些是筛选一切信息的底层价值观，优先级从高到低：

1. **深度优于广度** — 一个真正理解透彻的洞察，胜过十个浅尝辄止的信息
2. **原创优于衍生** — 第一手的创造和思考 > 对他人工作的总结和评论
3. **实践优于理论** — 有人真正用它构建了什么 > 只是概念论证
4. **反直觉优于确认** — 挑战既有认知的发现 > 强化已知结论的信息
5. **长期优于短期** — 3 年后仍然重要的 > 下周就过时的
6. **优雅优于复杂** — 用更少的东西解决更多的问题

## 筛选打分标准（详细版）

### 新颖性 (1-5)
- 5: 全新的范式/方法，从未见过
- 4: 已知方向的重大突破
- 3: 有意思的新组合或新应用
- 2: 增量改进
- 1: 旧概念换皮

### 深度 (1-5)
- 5: 有严谨的技术论证/实验数据/源码（来源为官方/arXiv/一手报告）
- 4: 有实质性的架构设计或深入分析（来源为可信媒体/知名技术博客）
- 3: 有一定深度但仍在表面，**或来源为二手聚合平台（CSDN/知乎/ithome 等）——二手聚合来源深度上限为 3 分**
- 2: 偏概述/入门
- 1: 只是标题党或新闻稿

> **来源可信度影响深度评分上限**（v0.2.0 新增）：二手聚合来源的内容，无论技术细节多丰富，深度上限为 3 分，因为无法验证原始数据的真实性。

### 实用性 (1-5)
- 5: 今天就能用，直接解决现有问题
- 4: 1-2 周内可以集成
- 3: 需要 1 个月的适配
- 2: 有启发但短期用不上
- 1: 纯学术/理论

### 趋势信号 (1-5)
- 5: 被 5+ 独立来源同时提到，明显趋势
- 4: 多个早期信号指向同一方向
- 3: 有一些关注但还未形成共识
- 2: 孤立事件
- 1: 可能是昙花一现

### 品味匹配 (1-5)
- 5: 完美符合核心审美原则 + Tier 1 领域
- 4: 符合核心原则 + Tier 2 领域
- 3: 部分符合，有一些有趣的角度
- 2: 边缘相关
- 1: 不在关注范围

Gary Marcus、Pieter Levels、Simon Willison 的内容单独计入**校准信号**类别，不计入通过率统计。

- 每次扫描最多收录 3 条校准信号（每个强制来源 1 条），超出部分降级为参考链接
- 通过率统计仅针对非强制来源的内容候选
- 校准信号在 filtered 列表中单独标注 `[校准信号]`，与普通 `[deep_dive]` / `[watch]` 区分

## deep_dive 触发条件（v0.2.2 新增）

进入 deep_dive 队列必须同时满足以下条件：

1. **总分 ≥21/25**
2. **新颖性 ≥4**
3. **实用性 ≥4 OR 趋势信号 ≥4**
4. **链接已独立验证**（非模型推断生成，必须来自搜索结果或直接访问确认）

不满足条件4的高分项目进入 **watch + 链接验证队列**，验证通过后可升级为 deep_dive。

成熟工具（项目年龄 >3 年，新颖性 ≤3）直接进入**立即使用队列**，不占用 deep_dive 资源。

**承诺延后项目优先级高于新候选**：每次 DIVE 开始前必须先检查承诺队列。

## 反模式（主动跳过的内容）

这些类型的内容无论分数多高都直接跳过：

- **炒作型**: "X will replace Y forever!" 没有技术论证的断言
- **列表型水文**: "Top 10 AI tools in 2026" 这种聚合内容
- **身份政治型**: 任何关于"AI 会不会取代程序员"的争论
- **Vaporware**: 只有 demo 视频没有源码/文档的项目
- **套壳项目**: 只是在别人 API 上包了一层 UI 没有创新
- **Token 驱动**: 主要靠发币/空投驱动的项目
- **二手聚合型**: 来源为 CSDN/知乎/腾讯新闻等平台且无法追溯原始来源的内容，直接降级，不进入 deep_dive（可作为线索追溯原始来源后重新评估）
- **冗余重复型**: 与同批次已收录内容高度重叠（>70% 信息重合）的条目，即使来源不同也跳过
- **链接验证失败型**: GitHub 项目链接无法直接访问时，不直接移除，执行'降级+追溯'流程——标记为'链接待验证'，尝试追溯正确仓库链接，追溯成功则重新评估，追溯失败才移除
- **跨批次重复型**: 同一项目在不同扫描批次中重复出现时，如果已有 deep_dive 报告（检查 memory/deep-dives/ 目录），则在 filtered 层直接标注'已分析，跳过'，不再进入 deep_dive 队列。如果仅有 watch 记录，可重新评估是否升级为 deep_dive（v0.2.0 新增）
- **URL 重复型**: 同一 URL 被分配给不同标题的多个条目时，两个条目均降级为 watch 并标注'URL 冲突，需重新验证'（v0.2.1 新增）
- **同账号幻觉传染型**（v0.2.3 修订）: 如果同一 GitHub 账号的某个项目被 Reviewer 确认为幻觉 URL 或链接验证失败，则该账号下的其他项目**触发独立链接验证**（而非自动降级）——验证通过则正常评估，验证失败才降级。**Reviewer 的纠错优先于规则自动触发**——如果 Reviewer 明确确认某账号项目为真实项目，应立即从可信度观察名单移除并重新评估。（原规则过于激进，导致 ruvnet/ruflo 被错误降级，v0.2.3 修订）
- **主条目交叉验证冗余型**: 同一项目的 GitHub 主条目和媒体分析文章（InfoQ/Medium 等）不得同时进入 filtered 列表——媒体分析文章应作为主条目的'来源可信度加分'而非独立条目（v0.2.2 新增）
- **过期来源型**（v0.2.3 新增）: Simon Willison Blog 等强制来源中，超过 7 天的文章不计入强制来源覆盖，Scout 必须继续搜索直到找到 7 天内的内容或明确标注'本日无新文章'。不接受以旧文章充数的强制来源执行。

## 扫描质量标准

_这是对扫描过程本身的约束，不是对内容的约束。_

- **最低采集量**: 每次扫描至少采集 **30 条**候选（20 条为绝对下限，30 条为目标），筛选后保留 6-9 条内容候选 + 最多 3 条校准信号（目标通过率 20-30%，**校准信号不计入通过率**）。低于 20 条的扫描视为失败，必须重新执行；20-29 条视为不完整扫描，需补充
- **分批执行建议**（v0.2.1 新增）：批次1覆盖 GitHub Trending 全部 25 个项目（目标 ≥15 条），批次2覆盖 HN Top 30（目标 ≥5 条），批次3覆盖专项来源（Simon Willison + Gary Marcus + Pieter Levels + 量化专项）。每批次独立记录，Evolve 阶段合并去重
- **强制覆盖源**（v0.2.3 更新）: 每次扫描必须覆盖：
  - GitHub Trending（**必须遍历全部 25 个项目**，目标 ≥15 条候选）
  - HN Top 30（≥5 条候选）
  - Simon Willison Blog（最新 3 篇，**只接受 7 天内文章**）
  - Gary Marcus Twitter/Substack（最新内容，**硬性阻断：缺失则扫描不完整，必须补充后才能进入 Analyst 阶段**）
  - Pieter Levels Twitter/Blog（最新内容）
  - **r/algotrading（目标 ≥3 条候选，v0.2.3 升为强制来源）**
  - 缺失任一视为不完整扫描
- **领域覆盖检查清单**（v0.2.0 新增，v0.2.3 更新）：每次扫描结束前，Scout 必须检查以下领域是否有至少 1 条候选：① AI Agent 编排（Tier 1，权重 10）② LLM 应用架构（Tier 1，权重 9）③ 量化/交易技术（Tier 1，权重 8，**必须有至少 1 条来自 r/algotrading 的主动扫描结果**）
- **英文源覆盖**: 每次扫描至少包含 2 个英文一手源（GitHub Trending、HN、Simon Willison 等）
- **来源可追溯**: 每条入选内容必须有可验证的原始来源链接
- **URL 独立验证**（v0.2.2 强化）：所有 GitHub 链接必须来自搜索结果或直接访问确认，不得由模型推断生成
- **去重检查**: 同一事件/项目的多个报道只保留来源最权威、信息最完整的一条；跨批次重复项目检查 memory/deep-dives/ 目录后标注'已分析，跳过'
- **跨批次合并**（v0.2.1 新增）：同一天不同批次的 filtered 条目必须在 Evolve 阶段做合并去重
- **REFLECT 规则提炼**（v0.2.2 新增）：每次 REFLECT 必须输出至少 1 条新规则或 1 条规则修订，不得输出空 changes 列表
- **承诺队列检查**（v0.2.3 新增）：每次 DIVE 开始前必须先检查承诺延后项目队列，优先处理承诺项目，再处理新的 filtered 候选
